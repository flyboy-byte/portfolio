HP LaserJet P1005 on Arch Linux
INSTALL • DIAGNOSE • OPEN-SOURCE FALLBACK • REPAIR LOG

The HP LaserJet P1005 is a host-based (“GDI / winprinter”) laser. It has no built-in page language and no network or IPP support — your PC rasterizes every page, and the printer needs its firmware re-uploaded every time it powers on. Two consequences: setup is fussier than a modern IPP printer, and there are two completely different driver paths you can take.

How to use this guide
  • Plan A (Sections 1–5) — install with HPLIP, HP's official stack (proprietary plugin + hpcups).
  • Sections 6–7 — diagnose and manage the queue when something's off.
  • Plan B (Section 8) — if HPLIP keeps fighting you, drop it and use the open-source foo2zjs / foo2xqx driver instead.
  • Section 11 — a personal repair log: real problems hit on this machine and what actually fixed them.
Read this first — P1005 quick facts
  • Pick one driver stack, not both. HPLIP and foo2zjs both want to own this printer; running them together causes blank pages. Remove one before installing the other.
  • Firmware loads on every power-on. If the attention light doesn't flash shortly after you power up or replug, firmware didn't upload and nothing will print.
  • usblp is driver-dependent. HPLIP talks to the device over libusb and generally does not want usblp bound; the classic foo2xqx hotplug historically needed the usblp lp node to push firmware. Match it to your stack.
  • Long game: CUPS is removing classic drivers/PPDs (Section 10). This printer has no IPP, so plan to wrap it in a "printer application" eventually.

> 1. Install with HPLIP (Plan A)

Plan A is HP's official stack: HPLIP plus its proprietary plugin, driving the printer through the hpcups filter. This one line pulls the printing core, HP driver, GUI tools, and dependencies via yay:

yay -S cups hplip hplip-plugin ghostscript python-pyqt5 python-dbus
python-pillow python-reportlab system-config-printer
sudo systemctl enable --now org.cups.cupsd
Note

On Arch the CUPS unit is cups.service (with cups.socket). org.cups.cupsd.service is just an alias, so either name works in systemctl commands.

> 2. Verify printer detection

Plug in the printer and check it's visible on USB. If missing, load the USB printer kernel module as a one-off check.

lsusb | grep -i hp
sudo modprobe usblp
lsusb
usblp caveat

Use usblp here only as a detection check. On Plan A (HPLIP) you generally do not want it permanently bound — HPLIP drives the device over libusb.

> 3. Install HP proprietary plugin (firmware + PPD)

The plugin provides the firmware and model driver the P1005 needs to function.

sudo hp-plugin -i
ls /usr/share/hplip/data/firmware/sihpP1005.dl
ls /usr/share/ppd/HP/HP_LaserJet_P1005.ppd.gz
This step causes most P1005 failures
  • hp-plugin -i downloads the plugin from HP's servers at install time. That download can fail, or pull a version that doesn't match your installed HPLIP — the single most common P1005 problem. See Section 6 and the repair log (Section 11).
  • The firmware file may be sihpP1005.dl or sihpP1005.dl.gz depending on HPLIP version. If the first ls is empty, try the .gz name.

> 4. Configure the printer

Use HPLIP's setup tool to register the printer — interactive or graphical mode (graphical needs PyQt).

sudo hp-setup -i
sudo hp-setup

> 5. Test the printer

Restart the printing service and send a test page or a simple text job.

sudo systemctl restart org.cups.cupsd
hp-testpage -p p1005
echo "Arch Linux HP P1005 test" | lp -d p1005
Note

Don't assume the queue is named p1005. Run lpstat -t (or lpstat -p) to see the real queue name and use that with lp -d.

> 6. When it’s not working — diagnose first

Before changing anything, find out what's actually failing. A P1005 job can show as "completed" while nothing prints — trust the logs over the queue status.

Read the logs

journalctl -u cups -b --no-pager
sudo cupsctl --debug-logging        # verbose logging on
sudo less /var/log/cups/error_log
sudo cupsctl --no-debug-logging      # turn it back off when done

Let HPLIP check itself

hp-check -t
hp-doctor

hp-doctor and hp-check flag a plugin version mismatch directly — the exact failure logged in Entry 1.

Common symptoms

SymptomFirst move
Job completes/fails but nothing printsCheck error_log / journalctl for HPLIP plugin errors. Plugin version mismatch? Fix per the repair log — or switch to Plan B.

Decision point

If the plugin mismatch keeps coming back after reinstalling (Entry 1), stop fighting HPLIP and go to Plan B (Section 8).

> 7. Manage & inspect the queue

Commands to view, manage, and debug printer queues.

lpstat -p -d
lpstat -v
lpstat -t
sudo lpadmin -x HP_LaserJet_P1005_2     # remove a queue
sudo systemctl restart org.cups.cupsd
http://localhost:631                     # CUPS web UI
hp-toolbox
hp-systray &

Rename a queue / set default

sudo lpadmin -p newname -E -v usb://HP/LaserJet%20P1005 \
    -m drv:///hp/hpcups.drv/hp-laserjet_p1005.ppd
lpoptions -d p1005

> 8. Plan B — open-source driver (foo2zjs / foo2xqx)

If HPLIP's plugin keeps breaking — version mismatches, failed downloads, jobs that vanish — the long-standing community answer for this exact printer is to drop HPLIP entirely and use the open-source foo2zjs package (from the AUR). It includes the foo2xqx driver for the P1005's QPDL protocol, loads the same firmware blob, but through its own udev hotplug rules — with no proprietary plugin to version-match.

Remove HPLIP first, then reboot

The two stacks collide. If no other HP device needs HPLIP, remove it and reboot so nothing of HPLIP's is still bound to the printer.

yay -Rns hplip hplip-plugin
sudo reboot

Then install the open-source driver:

yay -S foo2zjs

Make sure the P1005 firmware is present and the hotplug rule is installed so firmware uploads on power-on. If the package didn't fetch firmware automatically, grab it with the project's helper and install the hotplug rules:

./getweb P1005          # downloads sihpP1005.dl
sudo make install-hotplug
AUR packages vary

Exact steps depend on how the current AUR package is built — read its pacman post-install message / PKGBUILD. Some versions handle firmware and hotplug for you; others leave getweb to you.

Power-cycle the printer — the attention light should flash as firmware loads. Then add the queue, easiest via the CUPS web UI:

http://localhost:631     # Administration → Add Printer

Pick the local USB HP LaserJet P1005 and choose the foo2xqx PPD (listed as something like "HP LaserJet P1005 Foomatic/foo2xqx"). Or add it from the command line:

lpadmin -p P1005 -E -v usb://HP/LaserJet%20P1005 \
    -P /usr/share/cups/model/HP-LaserJet_P1005.ppd
lp -d P1005 /usr/share/cups/data/testprint
Blank pages?

Blank output almost always means firmware didn't load. Confirm the attention light flashed on power-on and that the hotplug rule fired. With foo2xqx's classic hotplug this is exactly where usblp matters — it may need loading so the lp node exists for the firmware push.

> 9. Keep it working — persistence

Power-cycle the printer and print again. Firmware re-uploads automatically on each power-on or replug — with HPLIP via its backend, with foo2zjs via the udev hotplug rule. No flash means no print.

dmesg | grep P1005
Snapshot a known-good config

Once a queue works, save the working CUPS config so you can restore it after an update breaks things:

sudo cp /etc/cups/printers.conf ~/printers.conf.good
sudo cp -r /etc/cups/ppd ~/cups-ppd.good

> 10. Background — CUPS is dropping classic drivers

Heads-up for the future. CUPS upstream has deprecated printer drivers, backends, and PPD files; a future release removes them and routes everything through IPP Everywhere and "Printer Applications." The early warning shows up when lpadmin prints "Printer drivers are deprecated and will stop working in a future version of CUPS."

The P1005 has no IPP at all, so when that lands, both Plan A and Plan B — as classic drivers — stop working as written. The path forward then is to run the driver inside a printer application (e.g. a Ghostscript- or HPLIP-based printer-app) that speaks IPP to CUPS. Not urgent today; just know that's the direction once the warnings start.

> Summary

Your Arch system now has: HPLIP + Plugin managing firmware and drivers, CUPS handling job queues, HP GUI tools fully functional, automatic firmware upload each time you plug in or reboot. You can now print, monitor, and manage the HP LaserJet P1005 like a native Linux printer.

If HPLIP ever becomes more trouble than it's worth, Plan B (foo2zjs) is the fallback, and Section 10 covers the longer-term CUPS migration.

> 11. Real Fixes Added Over Time

Running repair log for my own Arch + HP LaserJet P1005 setup. Not generic docs. Each entry = a real problem I hit and the fix that actually worked. Add a new entry every time something breaks and gets fixed.

Entry 1 — HPLIP plugin version mismatch after update/reinstall
Symptoms
  • HP LaserJet P1005 queue existed.
  • CUPS was enabled and active.
  • Printer had an HPLIP USB URI.
  • Jobs entered the queue but did not actually print.
  • The queue could become disabled after failed jobs.
  • Installed packages looked correct:
    • hplip 3.26.4
    • hplip-plugin 3.26.4
  • But CUPS logs showed HPLIP was still loading an older proprietary plugin binary.
Log pattern
validate_plugin_version() Plugin version[3.25.8] mismatch with HPLIP version[3.26.4]
Plugin version is not matching
m_Job initialization failed with error = 48
Invalid Library hanlder pLibHandler = NULL.
Cause

The installed package version did not prove the loaded plugin binary was correct. A stale older HPLIP plugin binary survived somewhere under the HPLIP plugin paths. HPLIP 3.26.4 refused to use the old 3.25.8 plugin, so the print job failed.

Fix

Remove and reinstall hplip-plugin via pacman/yay. Use find instead of shell globs so zsh does not fail with "no matches found."

Commands
sudo pacman -Rns hplip-plugin
sudo find /usr/share/hplip -path '*plugins*' -type f -name '*.so' -delete
yay -S hplip-plugin
sudo systemctl restart cups
sudo cupsenable <QUEUE_NAME>
sudo cupsaccept <QUEUE_NAME>
Verify the queue name
lpstat -t
Test print using the real queue name
lp -d <QUEUE_NAME> /usr/share/cups/data/testprint
Example from my setup
lp -d Classroom_P1005 /usr/share/cups/data/testprint
Important notes
  • Do not assume the queue is named p1005. Check lpstat -t.
  • Do not assume hp-plugin -i download mode works; HP's plugin downloader can fail.
  • Do not assume hplip-plugin being installed means the loaded binary is correct.
  • Do not use zsh globs like /usr/share/hplip/prnt/plugins/*.so unless the files are confirmed to exist. zsh may stop with "no matches found."
  • Do not delete duplicate queues until you confirm which queue actually works.
  • A CUPS job can appear completed or processed even when nothing prints, so check journalctl -u cups.
