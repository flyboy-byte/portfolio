Global Intent

This document captures a real Arch Linux install progression, not an idealized tutorial written after the fact by someone pretending every command worked the first time. The goal is to preserve the technical reasoning, failed paths, corrections, and final architecture so the install can be understood later as a reference.

The machine was an older Dell Latitude E5540. The target evolved during the process. The first idea was to dual boot pure Arch beside an existing EndeavourOS install. That shifted after filesystem resize issues and GRUB confusion made the dual-boot path less worth preserving. The final goal became cleaner: wipe the whole disk, install pure Arch, use BIOS/MBR because that is how the machine was already booting, encrypt the root filesystem with LUKS, keep /boot unencrypted, and use LXQt instead of a more manual Wayland setup.

The core principle was not "make the fanciest Arch install." The core principle was "make the layers match." Firmware mode, partition table, bootloader, encryption, initramfs, networking, desktop environment, and user permissions all had to line up. When they did not, the system failed in ways that looked random but were actually logical.

Abstract

This install started as a pure Arch dual-boot project on a Dell Latitude E5540 that already had EndeavourOS installed. The existing system had no EFI partition and appeared to be installed on a DOS/MBR partition table using legacy BIOS boot. The first plan was to shrink the EndeavourOS partition, create a new encrypted Arch partition, and let GRUB dual boot both systems.

That plan ran into ext4 tooling and partition-resize problems. A Mint live environment had an older e2fsprogs version that could not understand newer ext4 features from the Arch-family install. Later, the EndeavourOS partition appeared to have a mismatch between filesystem superblock size and physical partition size, likely caused by incomplete or mismatched shrink operations. GRUB also became confused because the system had bootloader files in more than one place, and the encrypted Arch /boot path created early-decryption issues.

The final resolution was to wipe /dev/sda, create a clean MBR layout, use a small unencrypted /boot partition, place the root filesystem inside a LUKS container, install GRUB for BIOS with i386-pc, and let the initramfs handle unlocking root. LXQt was chosen over labwc because the user wanted a complete lightweight desktop without manually assembling panels, applets, launchers, and session behavior.

1. Starting Condition: The Laptop Was Not Using UEFI

The first thing that mattered was not Arch. It was firmware mode.

The Dell Latitude E5540 was already running EndeavourOS, but the disk layout looked odd at first: there was only one ext4 partition and no visible EFI partition. That raised the question: was this really UEFI, or was it legacy BIOS?

The checks were simple:

ls /sys/firmware/efi

There was no EFI directory. Also, /boot had no efi folder. Then fdisk -l showed:

Disklabel type: dos

Fact: dos here means an MBR partition table.

Conclusion: the machine was booting in legacy BIOS mode, not UEFI.

That decision controlled the rest of the install. No EFI System Partition was needed. No systemd-boot. No UEFI boot entry management. GRUB would be installed to the disk itself:

grub-install --target=i386-pc /dev/sda

That command installs GRUB for BIOS/MBR boot. The target name i386-pc does not mean the installed OS is 32-bit. It is GRUB's BIOS platform target.

2. First Plan: Dual Boot Beside EndeavourOS

The original layout had EndeavourOS on /dev/sda1. The plan was to shrink that partition, create /dev/sda2, install Arch into it, and encrypt only the new Arch install.

The intended layout was roughly:

/dev/sda1    EndeavourOS existing ext4
/dev/sda2    LUKS container for Arch

This was technically possible. BIOS/MBR systems can dual boot just fine. GRUB in the MBR can present both systems. But the details got messy because the existing partition had to be resized, and resizing is not one action.

The important distinction:

  • e2fsck checks or repairs the filesystem.
  • resize2fs changes the size of the ext4 filesystem.
  • parted/GParted changes the partition boundary.

A filesystem can be smaller than the partition. A partition can also be accidentally made smaller than the filesystem, which is where the goblin math starts.

At one point, the filesystem had been shrunk or partially checked, but GParted still showed the whole partition. That made sense: resize2fs can shrink the ext4 filesystem inside the partition, but it does not change the partition table. The partition boundary still has to be moved separately with parted resizepart or GParted.

The intended command relationship was:

e2fsck -f /dev/sda1
resize2fs /dev/sda1 175G
parted /dev/sda
resizepart 1 175GB

The exact numbers changed during the process, but the principle was the same: shrink the filesystem first, then shrink the partition boundary.

3. Mint Live Environment Problem: Old e2fsprogs Versus Newer ext4 Features

The resize work was attempted from a Mint live boot. Mint saw errors when checking the EndeavourOS partition. Running e2fsck produced an unsupported feature message, including FEATURE_C12.

The installed Mint e2fsprogs version was 1.46. That was too old for some newer ext4 features used by the Arch/Endeavour filesystem. This did not automatically prove the filesystem was broken. It proved the tool was too old to safely handle that filesystem.

The important lesson: a live ISO is not automatically the right recovery environment. The age of userspace tools matters. Filesystems are not just "ext4 forever, same thing everywhere." Newer ext4 features can confuse older check/repair tools.

The Arch ISO was the better tool because it shipped newer filesystem utilities.

4. The Resize Path Became Untrustworthy

After more resizing attempts, e2fsck eventually reported that the superblock size and physical size were different. That likely meant the ext4 filesystem and the partition table were no longer describing the same size.

That was the point where the dual-boot plan stopped being worth it.

Working theory: the partition boundary and filesystem size got out of sync during the shrink attempts. This may have happened because one tool completed part of the operation while another failed or refused to continue.

Practical conclusion: stop trying to preserve the EndeavourOS install. Wipe the disk and rebuild cleanly.

That was the right call. Not because dual boot was impossible, but because the existing partition had become a bad foundation.

5. Wiping the Disk: What wipefs Does and Does Not Do

The decision was made to fully clean /dev/sda.

The command:

wipefs -a /dev/sda

This removes filesystem and partition signatures. It makes the disk appear blank to the OS.

It does not securely erase every sector. Old data may remain recoverable until overwritten. For reinstalling, this is fine. For selling or disposing of a disk, it is not enough.

Optional stronger first-sector cleanup:

dd if=/dev/zero of=/dev/sda bs=1M count=10 status=progress

That clears the beginning of the disk, including old bootloader and partition metadata. It is still not a full secure erase, but it helps avoid old signature weirdness.

6. Final Partition Design: MBR, Unencrypted /boot, Encrypted Root

The clean install used an MBR/DOS partition table because the laptop was already working in BIOS mode.

Partitioning was done with:

cfdisk /dev/sda

Choose: dos

Final layout:

/dev/sda1    512M or similar    Linux filesystem    /boot
/dev/sda2    rest of disk       Linux filesystem    LUKS root

For BIOS/MBR, neither partition strictly needs to be marked bootable for GRUB. GRUB installs to the MBR of /dev/sda. However, some older BIOS implementations expect at least one partition to have the bootable flag. Marking /dev/sda1 bootable is harmless and reasonable.

So the practical choice was:

  • /dev/sda1 bootable
  • /dev/sda2 not bootable

This layout is simple and robust. The separate unencrypted /boot avoids making GRUB decrypt the root volume before loading the kernel.

7. LUKS and Filesystem Setup

The encryption layer goes on /dev/sda2.

Commands:

cryptsetup luksFormat /dev/sda2
cryptsetup open /dev/sda2 cryptroot

After opening, the decrypted device appears as:

/dev/mapper/cryptroot

This mapper device is what gets formatted as the root filesystem:

mkfs.ext4 /dev/mapper/cryptroot

The boot partition is plain ext4:

mkfs.ext4 /dev/sda1

The order matters. Do not create LUKS on /dev/sda2 and then run mkfs.ext4 /dev/sda2. That would overwrite the LUKS container. The filesystem belongs inside the unlocked mapper.

Mounting:

mount /dev/mapper/cryptroot /mnt
mkdir /mnt/boot
mount /dev/sda1 /mnt/boot

At this point:

  • /mnt — future /
  • /mnt/boot — future /boot

This creates the intended boot model:

  • GRUB can read /boot without decrypting root.
  • The initramfs handles decrypting root.

8. Pacstrap Package Intent

The base install was not meant to be ultra-minimal. The intent was a practical base system that could boot, connect to networks, build AUR packages, and later install LXQt.

The package set discussed included:

pacstrap /mnt base base-devel linux linux-firmware \
grub cryptsetup os-prober \
nano vim zsh sudo \
networkmanager iwd dhclient \
sof-firmware linux-firmware-qlogic \
git curl wget unzip zip tar htop man-db man-pages less

Purpose:

  • base, linux, linux-firmware: core Arch system.
  • base-devel: required for makepkg and AUR building.
  • grub: BIOS bootloader.
  • cryptsetup: LUKS support.
  • nano, vim: editors.
  • zsh: preferred shell.
  • sudo: normal admin workflow.
  • networkmanager, iwd, dhclient: network stack options.
  • sof-firmware: audio firmware support on many laptops.
  • linux-firmware-qlogic: extra firmware coverage useful on some hardware.
  • git, curl, wget: package/build/recovery tools.
  • man-db, man-pages, less: offline documentation.
  • htop, unzip, zip, tar: basic system usability.

Since this was a Dell Latitude E5540, the most likely hardware support came from the kernel and linux-firmware. Intel wireless is usually handled by iwlwifi, and Intel graphics by the kernel's DRM/i915 stack. sof-firmware was included because audio firmware issues are common enough on laptops that it is cheap insurance.

9. Fstab and Chroot

After installing packages:

genfstab -U /mnt >> /mnt/etc/fstab

Expected idea:

UUID=<root-filesystem-uuid>  /      ext4  rw,relatime  0 1
UUID=<boot-filesystem-uuid>  /boot  ext4  rw,relatime  0 2

Then:

arch-chroot /mnt

From this point, commands affect the installed system rather than the live ISO.

10. Initramfs: The encrypt Hook Is Mandatory

The initramfs must know how to unlock the LUKS root device. This is controlled by /etc/mkinitcpio.conf.

The hook line seen earlier contained modern Arch defaults:

HOOKS=(base udev autodetect microcode modconf kms keyboard keymap consolefont block filesystems fsck)

For encrypted root, encrypt must be inserted after block and before filesystems.

Correct concept:

HOOKS=(base udev autodetect microcode modconf kms keyboard keymap consolefont block encrypt filesystems fsck)

The important rule:

block → encrypt → filesystems

block detects block devices. encrypt opens the LUKS device. filesystems mounts the root filesystem after it exists.

The keyboard-related hooks are also useful because the passphrase is entered early in boot.

After editing:

mkinitcpio -P

Warnings about possibly missing firmware may appear. If image generation succeeds, those warnings are usually not fatal. They often mean optional firmware for hardware variants is not present.

11. GRUB Configuration for BIOS/MBR with LUKS Root

Because /boot is unencrypted, GRUB does not need GRUB_ENABLE_CRYPTODISK=y.

This was a key correction from the earlier failed setup. When /boot was inside the encrypted root, GRUB tried to decrypt the LUKS container itself and produced passphrase prompts at the GRUB stage. That made troubleshooting messy. With separate /boot, GRUB should not decrypt anything. It should only load the kernel and initramfs.

Find the LUKS UUID:

blkid /dev/sda2

Use the UUID from /dev/sda2, where TYPE="crypto_LUKS" appears. Do not use the UUID from /dev/mapper/cryptroot.

Edit:

nano /etc/default/grub

Set:

GRUB_CMDLINE_LINUX_DEFAULT="loglevel=3 quiet cryptdevice=UUID=<sda2-luks-uuid>:cryptroot root=/dev/mapper/cryptroot"

No dual boot means os-prober is not necessary. It can be installed, but it does not need to do anything.

Install GRUB:

grub-install --target=i386-pc /dev/sda
grub-mkconfig -o /boot/grub/grub.cfg

Expected result:

Installation finished. No error reported.
Found linux image: /boot/vmlinuz-linux
Found initrd image: /boot/initramfs-linux.img
done

If GRUB asks for a LUKS password before the kernel loads, something is wrong for this layout. With unencrypted /boot, the LUKS prompt should come later from the initramfs.

Expected boot:

  • GRUB menu appears.
  • Kernel/initramfs loads.

That boot behavior confirms the layering is correct.

13. Desktop Decision: labwc Was Cleaner, LXQt Was Smarter for This Job

The original desktop idea was labwc. That made sense technically. labwc is a Wayland stacking compositor inspired by Openbox. Combined with Waybar, it can make a clean, lightweight desktop.

The problem was not whether labwc was good. The problem was how much manual assembly it required.

A labwc setup would likely need:

  • labwc for window management.
  • waybar for panel.
  • fuzzel, wmenu, wofi, or rofi-wayland for launching apps.
  • mako for notifications.
  • nm-applet for network tray.
  • pavucontrol or audio applets for sound.
  • manual autostart configuration.
  • manual keybinds and menu configuration.

That was more fiddling than desired.

LXQt became the better target because it gives a complete lightweight desktop without turning the install into a desktop-plumbing side quest.

Install after boot:

sudo pacman -S lxqt sddm xorg xorg-xinit xorg-drivers

Enable graphical login:

sudo systemctl enable sddm

Audio stack, preferably PipeWire:

sudo pacman -S pipewire pipewire-pulse wireplumber alsa-utils pavucontrol

Network tray:

sudo pacman -S network-manager-applet

Useful desktop tools:

sudo pacman -S chromium terminology pcmanfm-qt ark gvfs

This gives a practical desktop with panels, settings, file manager, session management, and tray behavior without hand-building the whole shell.

14. Networking Issue: Connected with iwctl but DNS Failed

At one point, Wi-Fi was connected through iwctl, but hostnames would not resolve. That means the Wi-Fi link was probably up, but DNS was not working.

Diagnostic sequence:

ip a
ping -c 3 1.1.1.1
ping -c 3 archlinux.org
cat /etc/resolv.conf

Interpretation:

  • If ping to 1.1.1.1 works but archlinux.org fails, DNS is broken.
  • If neither works, routing/DHCP/link is broken.

Temporary DNS fix:

echo "nameserver 1.1.1.1" > /etc/resolv.conf

Long-term, NetworkManager should own the connection and DNS handling.

15. User Account Issue: useradd logan Was Incomplete

After boot, a normal user was needed to build yay. makepkg refuses to run as root, correctly. AUR builds should not run as root because PKGBUILDs execute build scripts.

The first attempt:

useradd logan

This created a user but did not create /home/logan.

Correct command:

useradd -m -G wheel -s /bin/zsh logan
passwd logan

Meaning:

  • -m creates the home directory.
  • -G wheel adds admin group membership.
  • -s /bin/zsh sets the login shell.

Enable sudo:

EDITOR=nano visudo

Uncomment:

%wheel ALL=(ALL:ALL) ALL

Switch into the user:

su - logan

Build yay:

git clone https://aur.archlinux.org/yay.git
cd yay
makepkg -si

That creates a proper user workflow: root for system recovery, normal user for daily use and AUR builds.

Final Boot Flow

  1. Power on.
  2. BIOS loads GRUB from MBR.
  3. GRUB reads /boot on /dev/sda1.
  4. GRUB loads Linux kernel and initramfs.
  5. Initramfs sees cryptdevice=UUID=<sda2 UUID>:cryptroot.
  6. Initramfs asks for LUKS passphrase.
  7. cryptsetup opens /dev/sda2 as /dev/mapper/cryptroot.
  8. Kernel mounts encrypted root as /.
  9. System boots to login or SDDM.

This is not the most exotic encrypted Linux setup possible. It does not use encrypted /boot, Secure Boot, TPM unlock, detached LUKS headers, or measured boot. Those can be valid, but they add complexity. For an older BIOS laptop, this design is strong because it is repairable and understandable.

17. Main Lessons

The big lesson was that Arch is not hard because the commands are magical. It is hard because the system exposes every layer.

The Install Failed When Layers Were Mixed

  • Mint tools were too old for newer ext4 metadata.
  • Filesystem size and partition size got out of sync.
  • GRUB config and actual bootloader location did not clearly match.
  • Encrypted /boot made GRUB responsible for decryption.
  • Dual boot added complexity while the old partition was already suspect.

The Install Worked When the Layers Were Simplified

  • BIOS mode matched MBR.
  • GRUB installed to /dev/sda.
  • Plain /boot let GRUB stay simple.
  • LUKS protected root.
  • mkinitcpio handled unlock.
  • LXQt avoided manual desktop assembly.
  • A real user account solved makepkg/yay workflow.

That is the clean engineering takeaway: do not make every component clever at the same time.
